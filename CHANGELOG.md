# Changelog

本项目遵循语义化版本（SemVer）。

## v3.5.0 - 2026-03-30

### 版本总述

`v3.5.0` 是小锤子监控台进入下一阶段产品化整理的一次正式版本升级。

这次升级不只是界面继续修一修，也不只是补几个小 bug。它覆盖了首屏启动链路、服务端首次快照时序、会话与 ACP 的业务表达、卡片状态体系、局部异常反馈、响应式表现以及浏览器真实环境回归。换句话说，`v3.5.0` 做的不是“让页面再好看一点”，而是“让这个监控台在真实使用里更稳、更像产品、更像一个可以长期开着看的面板”。

如果说 `v3.0.0` 解决的是结构问题，把它从“平铺数据板”升级成“总览 / 检修双视图”；那么 `v3.5.0` 解决的更多是成熟度问题：首屏节奏是否自然，状态语义是否完整，异常时是不是还说人话，窄屏和手机上是不是还能稳住。

### 首屏启动编排重做

本次在 `useMetrics.ts` 与 `metrics-state.ts` 中重写了首轮数据进入方式：

- 首次进入页面时，并行拉取 `metrics / acp-workflow / task-flow / task-events`
- 使用 `allSettled` 协调首轮资源返回
- 只有首轮快照完成后，才继续进入实时 WebSocket 链路
- 对外暴露 `bootState`、`resourceStates`、`refreshAll`

这项改动解决的核心问题，是过去首屏体验里经常出现的“页面刚打开像离线”“其实只是数据还没回来，但视觉上像出故障”的错觉。

`v3.5.0` 之后，页面会明确经历一个可理解的节奏：

- 正在连接监控源
- 同步首轮快照
- 数据开始进入
- 进入实时态

用户能直接感知到的变化，是首屏更稳，不再一上来就被离线语义吓一跳，也不需要靠猜去判断现在到底是没数据、网络抖了，还是页面真的坏了。

### 服务端首次快照提前

本次在 `server.ts` 中把第一次快照采集提前到服务启动后立即进行，不再保留此前多秒级的等待窗口。

这件事看起来像是一个技术时序调整，但它的产品意义非常明确：

- 页面能更早拿到第一批有效数据
- 首屏空档更短
- “刚打开什么都没有”的时间明显减少
- 启动体验更接近真正的监控产品，而不是“服务起来了，但还要等一会儿它才开始工作”

对于运行态监控来说，这种首批数据出现速度的改善，会直接影响用户对整个产品可靠性的第一印象。

### 业务命名与展示层统一

本次在 `dashboard-presenters.ts`、`format.ts`、`resource-state-copy.ts` 中补齐了一层更像产品的展示表达，并进一步整理了：

- `dashboard-task-summarizer`
- Cron 任务名回映
- 微信直连会话识别
- ACP 会话展示层
- ACP 指标口径文案

这项升级要解决的问题，是监控台过去某些区域仍然保留太多底层字段气味：看得懂的人能勉强猜，但不够像一个真的面向人使用的产品界面。

`v3.5.0` 之后：

- 会话名称更容易读
- Cron 任务不再只是原始 key
- ACP 区域不再只给一串技术标识
- 关键指标的文案更接近业务含义

这不是“把工程信息藏起来”，而是在不牺牲排障能力的前提下，把它翻译成更适合阅读和判断的信息层。

### 卡片状态体系统一

本次在 `CardStateNotice.tsx`、`App.tsx`、`Header.tsx`、`AttentionCard.tsx`、`SystemSummaryCard.tsx` 以及多张核心卡片中，统一补齐了：

- `loading`
- `error`
- `stale`
- `retry`

这项改动的意义在于，监控台终于不再只在“理想状态”下成立。

过去很多卡片在数据未到、局部失败、数据陈旧时，要么看起来像空白，要么只能从零散文案里猜。现在不同卡片在非正常状态下会有更统一的提示语义和展示方式，用户能更快判断：

- 这张卡是在加载
- 这张卡是接口暂时不可用
- 这张卡显示的是最近一次快照，但实时链路正在波动
- 我现在可以点哪里重试

这类状态体系补齐之后，页面整体观感会明显更成熟，因为它开始认真对待“不完美运行”的现实。

### 异常反馈与重试能力增强

这次版本特别强调“异常时也要体面”。

在真实回归之外，我们额外做了故障注入验证：人为让 `task-events` 返回 `503`，并禁用实时链路。页面最终能够正确展示：

- “任务事件暂不可用”
- 明确的局部错误提示
- `重试` 按钮

这件事的价值，不在于做出一个红色提示框，而在于监控台开始具备真正的产品级异常反馈能力：

- 告诉用户是哪一块出了问题
- 不把局部问题夸大成整体崩溃
- 在合适的位置给出恢复动作

这会让整个产品从“正常时看着不错”进化成“波动和失败时也还能用”。

### 多视口与移动端体验继续完善

本次版本继续围绕真实浏览器表现做了收尾和回归：

- `1600x1200` 桌面宽屏硬刷新通过
- `390x844` 手机宽度硬刷新通过
- `Overview / Inspect` 两页都完成复查
- 手机宽度下 `documentElement.scrollWidth = 375`、`innerWidth = 390`
- 无横向溢出
- 正常路径下 `console` 0 报错
- 核心接口 `metrics / acp-workflow / task-flow / task-events` 全部返回 `200`

用户能感知到的变化，是界面在桌面和手机之间切换时更稳了，不会轻易出现“宽屏还行，手机一看就散”“任务流和长文本把版面撑坏”的情况。

### 整体观感、节奏和细节更成熟

如果把这次版本用一句话概括，可以说：`v3.5.0` 让小锤子监控台更像一个真的产品，而不是一个慢慢长出来的工程面板。

变化不只在视觉上，也在交互节奏上：

- 首屏有启动语义
- 局部状态有层级
- 会话命名更可读
- 异常反馈更有边界
- 窄屏表现更稳定

这些点单看都不算“惊天动地”，但放在一起，会显著提高这个监控台的可长期使用性。

### 验证与回归

本次版本已完成以下验证：

- 构建通过
- TypeScript 校验通过
- 浏览器真实环境回归通过
- `Overview / Inspect` 桌面与手机宽度复查通过
- 正常路径下 `console` 无新增报错
- 正常路径下 `network` 关键接口返回正常
- 故障注入回归通过，局部错误态与重试入口工作正常

### 已知后续方向

`v3.5.0` 完成了这一轮最重要的成熟度补齐，但仍有几个值得继续推进的方向：

- ACP 与普通 sessions 的跨资源关联仍可继续加强
- ACP 命名仍可进一步细化到项目级 / 工作区级语义
- 重试入口目前以统一刷新为主，后续可细化成按资源粒度重试
- 某些被压缩展示的长文本，后续可继续补“展开 / 查看全文”之类更完整的交互
- 首屏之外的深层检修体验，仍可继续做更多半自动诊断提示

## v3.0.0 - 2026-03-25

### 里程碑
- 完成一次面向产品结构的重大升级：监控台从“平铺式实时看板”升级为“运营驾驶舱 + 检修入口”。
- 明确与 OpenClaw 官方 Dashboard 的职责分工：3210 负责上层运营视图与行动摘要，18789 负责底层系统真相与原厂诊断。

### 新增
- 新增总览 / 检修双视图切换，支持按场景切换日常查看和深入排查。
- 新增 AttentionCard，用规则化方式汇总“当前最值得关注”的事项。
- 新增 SystemSummaryCard，集中展示系统健康、连接状态和摘要级运行信息。
- 新增 AutomationStatusCard，接入自动化任务状态聚合结果。
- 新增 OfficialDashboardLinksCard，提供官方 Dashboard 深链入口。
- 新增 AcpWorkflowSummaryCard，在总览页展示 ACP 工作流摘要。
- 后端新增 `automation-status.ts` 与 automation / officialDashboard 数据输出结构。

### 增强
- `SessionsCard` 升级为“会话工作台”，增加优先级排序、风险标记和摘要条。
- `TaskLogCard` 升级为“任务时间线”，增加持续时长、最近活动时间和需关注状态识别。
- `AcpWorkflowCard` 升级为“ACP 工作流检修”视图，增加阶段、时长、慢工具、开始/更新时间等更多诊断细节。
- `App.tsx` 重构页面布局，按总览 / 检修组织现有卡片与新卡片。
- `format.ts`、`types.ts` 扩展格式化函数与类型定义，支持自动化状态、官方链接与持续时长展示。

### 样式与体验
- 前端样式进行大规模重构，新增双视图 tabs、总览/检修布局、摘要卡、风险 pill、自动化区块和更多响应式细节。
- 首页信息密度从“平铺卡片”调整为“先判断，再深入”的结构。
- 检修视图增强了会话、任务和 ACP 的可读性，更适合排查与回看。

### 说明
- 本次版本属于结构级大升级，变更范围覆盖前后端数据结构、页面布局与信息分层。
- v2.x 时代的“宽屏浏览器下偶发卡片大小与风格错位”问题已通过新布局体系明显缓解，但后续仍可继续做视觉细节打磨。
- README 展示位已切换为 v3 截图结构：桌面总览、桌面检修、手机总览；截图文件需按新命名补齐后推送。
